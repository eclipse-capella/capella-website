Components Exchanges

Engineering Data

Major Kinds of Engineering Data > SOLUTION Description > PHYSICAL ARCHITECTURE > Components Exchanges

A (behavioral) component exchange is an interaction between two components likely to exchange some items (information, signal, material, torque…). It realises functional exchanges between functions allocated to source and target components.

Activities producing the Data

  • Allocate component exchanges to physical links
  • Identify Architectural alternatives
  • Select and finalise the Physical Reference Architecture
  • Check each engineering/architecting decision against physical viewpoints
  • Define interfaces and exchanges between behavioural components
  • Select and validate best compromise Architecture
  • Promulgate Reference Architecture
  • Define Components Modes & States
  • Explore Solution Space & Alternatives
  • Deduce scenario allocation to components

Activities using the Data

  • Check against architecture drivers
  • Deduce scenario allocation to components
  • Allocate component exchanges to physical links
  • Identify Architectural alternatives
  • Validate each architecture against operational and functional/non functional need
  • Perform Speciality or Discipline Analyses
  • specify integration & test means
  • Build PIDS documents (Prime Item Development Specification) and subsystems, SW, HW specifications (SSS, SRS, HRS) including interfaces
  • Check each engineering/architecting decision against physical viewpoints
  • Build Product Breakdown Structure
  • Ensure consistency between Solution Architecture and product Variability
  • Optimise contents & order of integration steps
  • Define customer deliveries based on operational capabilities
  • Validate conformance of each candidate architecture against design expectations incl. viewpoints
  • Build SSDD document (System/Segment Design Document)
  • Define physical links and ports between implementation components
  • Select and validate best compromise Architecture
  • Finalise requirements based on physical architecture
  • Build ICD document (Interface Control Document)
  • Define IVVQ scenarios based on need analysis
  • Formalise Component development & integration contract
  • Perform multi-Disciplines Trade-off
  • Perform finer-grain analyses (e.g. per viewpoint)
